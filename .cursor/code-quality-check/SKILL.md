# 代码质量检查

在每次代码编写或修改完成后，按以下清单逐项检查，并向用户报告检查结果。

## 执行时机

当你完成以下操作后，**自动执行**此检查：
- 新增功能代码
- 修改现有代码
- 重构代码
- 修复 bug

## 检查清单

### 1. 文件与结构规范

- [ ] 单个文件不超过 500 行，类型定义多时拆分到独立 `.d.ts` 或 `types.ts`
- [ ] 遵循单一职责原则，每个文件/组件只做一件事
- [ ] 通用工具方法放在统一的工具文件中，不重复定义

### 2. TypeScript 规范

- [ ] 接口返回值使用可选链 (`?.`)
- [ ] 可能为空的数组使用 `|| []` 赋默认值，避免空引用
- [ ] 类型定义完整，避免 `any`
- [ ] 简单的 `if/else` 使用三元表达式

### 3. Vue 组件规范

- [ ] 组件结构清晰：`<script setup>` + `<template>` + `<style scoped>`
- [ ] 不多加额外的 DOM 层级，不需要的地方不套额外的 `div`
- [ ] `props` 和 `emits` 类型定义完整
- [ ] 响应式变量命名合理

### 4. 样式规范

- [ ] CSS 尽量精简，能一行实现的不用两行
- [ ] 不添加多余的类名
- [ ] 主色调和样式与项目其他页面和主题保持一致
- [ ] 使用 `scoped` 避免样式污染

### 5. 性能检查

- [ ] 函数无过大开销，注意循环中的计算复杂度
- [ ] 所有涉及计算的逻辑使用 `mathjs` 处理
- [ ] 大列表使用虚拟滚动或分页
- [ ] 组件合理使用 `v-memo`、`computed` 缓存计算结果
- [ ] 避免不必要的响应式包装

### 6. 安全检查

- [ ] 无 `v-html` 直接渲染用户输入
- [ ] API 请求参数经过校验
- [ ] 敏感信息不硬编码在前端代码中

### 7. 异常处理

- [ ] 不使用 `try-catch`，遵循项目统一的异常处理规范
- [ ] API 调用有错误状态处理
- [ ] 边界条件已处理（空数据、加载中、错误状态）

### 8. 注释与可读性

- [ ] 注释简洁，只解释非显而易见的逻辑
- [ ] 无多余的叙述性注释（如 "导入模块"、"定义函数"）
- [ ] 变量/函数命名自解释

### 9. 代码风格（Prettier）

- [ ] 使用分号
- [ ] 使用双引号
- [ ] 行宽不超过 150 字符

## 输出格式

检查完成后，使用以下格式简要报告：

```
## 代码质量检查结果

**通过项**: X/总数
**问题项**:
- [严重] 具体问题描述 → 建议修复方式
- [建议] 具体问题描述 → 优化方向

**总结**: 一句话总结代码质量状况
```

只报告发现问题的检查项，全部通过时输出"全部通过，代码质量良好"即可。

## 重要原则

- 检查应**快速且不阻塞**用户工作流
- 只报告**实际存在**的问题，不要凭空捏造
- 严重问题主动提出修复建议，建议类问题让用户决定是否处理
- 不要自作主张修改用户没有提到的代码
